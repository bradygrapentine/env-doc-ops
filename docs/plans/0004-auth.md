# Plan 0004 — Real auth (email + password)

Backlog: **B-040** · Priority **P1** · Effort **M**

## Goal

Add email + password authentication. Every project becomes scoped to a user. The first user adopts any pre-auth projects so the existing demo data isn't orphaned.

## Stack

- **Auth.js v5** (`next-auth@beta`) with the **Credentials** provider — no external email service.
- **bcryptjs** for password hashing (pure JS, no native build).
- Sessions stored in JWTs (Auth.js default). No DB session table.

## Schema

- New `users(id, email UNIQUE, passwordHash, name, createdAt)` table.
- `projects.userId TEXT` column, nullable on the existing rows; backfilled on first signup; new projects always get a `userId`. We don't add a `NOT NULL` constraint to keep the migration trivial — the route layer enforces ownership.

## Decision rules

- Password: ≥8 characters. No complexity rules for V1.
- Sessions: JWT, default 30-day max age (Auth.js default).
- First-user adoption: when a user signs up, if there are any projects with `userId IS NULL`, claim them all for that user. Documented as a one-time bootstrap.
- A signed-in user only sees, can edit, or can delete projects where `projects.userId = session.user.id`.
- Reports inherit ownership through `reports.projectId → projects.userId`. No new column on reports.

## Routes & pages

- `src/app/api/auth/[...nextauth]/route.ts` — Auth.js handlers.
- `src/app/signup/page.tsx` — email + password + name form. Creates user, signs them in, redirects to `/`.
- `src/app/signin/page.tsx` — email + password form. On success, redirects to `/` (or `?callbackUrl`).
- `src/app/signout/route.ts` — POSTs to Auth.js signout, redirects to `/signin`. (Or just expose a signout button that calls Auth.js `signOut()` from a client component — pick this simpler path.)

All other routes and pages require a session. Unauthenticated requests:

- API routes return 401.
- Page handlers redirect to `/signin?callbackUrl=...`.

## Test strategy

Going through Auth.js end-to-end in unit tests is heavy. Instead:

- `src/lib/auth.ts` exports `getSessionUserId()` that reads the session in normal mode, or reads `process.env.AUTH_TEST_USER_ID` if set. **Test-only backdoor**, clearly commented.
- All tests set `AUTH_TEST_USER_ID` to a freshly-created user's id, then exercise the route handlers as before.
- A small set of tests verify the auth flow itself (signup creates user + claims orphans, signin verifies password, route returns 401 with no session).

## Acceptance criteria

1. `npm run check` green.
2. `/signup` and `/signin` work end-to-end against a running server.
3. Signed-out user hitting `/`, `/projects/...`, `/reports/...`, or any API route gets redirected (pages) or 401 (API routes).
4. Two users created sequentially can't see each other's projects.
5. The pre-existing demo project (if any) is owned by the first user who signs up.
6. All previously passing tests still pass once `AUTH_TEST_USER_ID` is wired in.

## Out of scope (followups)

- Email verification.
- Password reset / change-password.
- "Remember me" toggle / configurable session length.
- OAuth providers.
- Rate limiting on signin / signup.
- Account deletion / settings page.
